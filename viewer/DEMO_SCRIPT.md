# AgentOS Viewer — Demo Script
### Meeting Walkthrough with Live Mock Events

---

## Before You Start

Make sure both servers are running:

```bash
# Terminal 1 — backend relay with mock events
cd viewer/server && MOCK_EVENTS=1 node server.js

# Terminal 2 — frontend dev server
cd viewer/viewer-app && npm run dev
```

Open **http://localhost:5173** in your browser. You should see events flowing in within a few seconds.

---

## 1. Opening — What Is This Dashboard?

> *"What you're looking at is a real-time process viewer for AgentOS — a sandboxed runtime for AI agents. The problem we're solving is visibility: when an AI agent is running autonomously, you need to know exactly what it's doing, what network connections it's making, and whether anything suspicious is happening — all in real time."*

> *"The dashboard connects to a WebSocket relay server that sits between the agent runtime and the browser. Any event the agent or the kernel produces — tool calls, network connections, security decisions — gets pushed here live."*

## 2. The Header Bar

**Point to the top bar.**

> *"At the top we have the AgentOS header. On the left is the brand. On the right is the WebSocket connection status — right now it shows 'ws connected' with a green indicator, meaning we have a live feed coming in."*

If a bar turns red, it means a blocked connection was detected in that window. It gives you an at-a-glance sense of how busy the agent is and whether anything is being flagged."*

## 3. The Sidebar — Control Panel

**Point to the left sidebar.**

> *"On the left is the Control Panel sidebar. It has a few sections:"*

- **Live / Offline indicator** — mirrors the WebSocket status, plus a live uptime counter showing how long this session has been running.

- **Views** — two navigation tabs: **Events** (the default detailed view) and **Workflow** (the graph view we'll show shortly). Click between them here.

- **Agents** — as agents connect and produce events, they appear here automatically. You can click an agent to filter the entire dashboard to just that agent's events. The coloured dot next to each agent shows its current state — pulsing green means running, grey means idle, amber means completed, red means crashed.

- **Quick counts** — a live tally of tool calls made, connections allowed, and connections blocked for the selected agent.

## 4. The Events Tab — Main View

**Make sure you're on the Events tab.**

### 4a. Stat Rings

**Point to the three arc gauges.**

> *"Just below the header we have three animated stat rings — semi-circle arcs that split to show a ratio."*

- **Connections** — left (green) side is allowed connections, right (red) side is blocked. The centre shows the raw allow:block count. As the mock demo runs you'll see this shift when the blocked connection fires.

- **Agent Activity** — shows total tool calls made. The left (blue) arc is safe calls, the right (amber) arc is calls flagged as potentially injected. This lets you see at a glance what proportion of the agent's network activity looks suspicious.

- **Session** — shows total session uptime in the centre, with clean events vs. detected threats on the arc. Purple is clean activity, red is threats.

> *"All three rings animate smoothly as new data comes in — watch the split point move when the blocked event fires."*

### 4b. Session Flow (MiniFlow)

**Point to the horizontal node chain.**

> *"Below the rings is the Session Flow bar — a compact linear representation of the agent's current session. Each node is a key event type:"*

- **▶ Start** — the session kicked off
- **Task** — the user gave the agent an instruction
- **Fetch** — the agent made a tool call (fetched a URL)
- **Result** — the tool call returned
- **Answer** — the agent produced its final response
- **Done** — the session completed cleanly

> *"Nodes are connected by animated arrows. If a tool call was flagged as blocked or suspicious, that node turns red and shows 'Blocked' instead. The last node pulses with a live indicator while the session is still running. If there are too many steps to show, the middle collapses to show '+ N more' so the bar never overflows."*

### 4c. Agent Events Panel (Left)

**Point to the left panel labelled "LLM — Agent events".**

> *"On the left we have the Agent Events panel — this is the LLM layer. Every event generated by the agent itself appears here as a card:"*

| Card type | What it means |
|---|---|
| **Session initialised** | The agent process started — shows PID and launch mode |
| **Task received** | The user's instruction was passed to the agent |
| **Agent fetched a URL** | The agent called its `fetch_url` tool, showing which URL |
| **Fetch succeeded / failed** | The result of the fetch — status code, character count |
| **Agent responded** | The agent produced its final answer |
| **Session completed** | The agent exited cleanly with exit code 0 |

> *"Each card has a timestamp on the right down to the millisecond. If there are more than 6 cards, a '+N more' button appears — click it to expand the full history. The panel auto-scrolls to the latest event without jumping when you're reading older cards."*

> *"If a tool call is identified as an injection target — meaning the kernel blocked the connection it triggered — that card gets a red highlight border to draw your eye to it."*

### 4d. The Alert Banner

> *"When a blocked connection is detected and linked back to a tool call, a red alert banner animates in at the top of the Agent Events panel. It tells you which hostname was blocked and why — for example 'Injection attack blocked by kernel — evil.com (no policy match)'. It auto-dismisses after 5 seconds, or you can close it manually."*

**Wait for the mock sequence to reach the blocked event and point to the banner when it appears.**

### 4e. Kernel / Network Events Panel (Right)

**Point to the right panel labelled "KERNEL — Network events".**

> *"On the right is the Kernel Events panel — this is the eBPF layer, sitting below the agent in the OS kernel. Every network connection the agent process attempts is intercepted here and shows up as a card:"*

| Card type | What it means |
|---|---|
| **Connection attempt** | The agent's process opened a socket — shows destination IP, port, and hostname |
| **Connection permitted** | The kernel allowed the connection — it matched the agent's `allowed_hosts` policy |
| **Connection refused by kernel** | The kernel blocked the connection — the hostname was not in the policy allowlist |

> *"This panel is completely independent of what the agent reports about itself — it's the kernel's ground truth. Even if a malicious prompt tricks the agent into trying to reach an external server, the kernel catches and blocks it here regardless."*

## 5. The Connection Timeline

**Scroll down or point to the timeline strip below the two panels.**

> *"Below the event panels is the Connection Timeline — a 60-second sliding window showing every allowed and blocked network connection as a dot on a horizontal axis."*

- **Green dots** = allowed connections, plotted at the moment they occurred
- **Red dots** = blocked connections, with a small vertical spike above them to make them visually distinct
- The **"now" line** on the right edge sweeps events leftward as time passes — older events fade off the left edge after 60 seconds

> *"Hover over any dot to see a tooltip with the hostname, whether it was allowed or blocked, and how many seconds ago it happened. This gives you a quick spatial sense of connection density and whether blocks are clustering in time."*

## 6. The Threat Gauge

**Point to the gauge component below the timeline.**

> *"At the bottom is the AI-powered Threat Gauge. This is driven by an OpenAI-backed analyser running server-side — it periodically reads the last 20 events and produces a structured threat assessment."*

- The **semi-circle gauge** has four colour zones: green (Low) → amber (Medium) → red (High) → deep red (Critical). A needle points to the current threat level.
- The **active arc** fills from the left up to the needle position in the appropriate colour.
- To the right of the gauge you'll see a **plain-English summary** of what the analyser detected, a **list of specific concerns**, and a **recommendation**.
- A small **"analysed N seconds ago"** label tells you how fresh the assessment is.

> *"In this demo the OpenAI key isn't set so the gauge shows 'awaiting first analysis', but in a real deployment it would update every time the agent produces a batch of events."*

## 7. The Workflow Tab

**Click "Workflow" in the sidebar.**

> *"Now let's switch to the Workflow tab — this is the graph view."*

> *"The entire content area is taken over by an interactive node graph that reconstructs the agent's session as a visual flow diagram. There are no stat rings up top — the graph gets all the space."*

### 7a. Session Summary Bar

**Point to the summary strip at the top of the graph.**

> *"At the top of the graph there's a session summary strip showing four numbers: total duration of the session, number of logical steps, number of external requests made, and how many were blocked. It updates live as events come in."*

### 7b. Main Chain (Left Column)

> *"The main vertical chain on the left represents the agent's logical flow:"*

- **▶ Session Started** circle — the beginning of the session, with the agent name and launch mode
- **Task Given** card — shows the user's actual instruction in quotes
- **Fetching URL** card — shows which hostname the agent is reaching out to, and the result below it (waiting / succeeded / blocked)
- **Agent Answer** card — the agent's final response, first sentence shown as a summary
- **✓ Task Complete** circle — the session ended successfully

> *"Between each pair of nodes there's a time delta label — for example '0.8s' — showing how long that step took. This makes it easy to spot where the agent spent most of its time."*

> *"The edges change colour based on outcome: a green-tinted edge leads into a successful fetch, a red animated dashed edge leads into a blocked one."*

### 7c. Kernel Nodes (Right Column)

> *"Branching off to the right of each tool call card are the kernel outcome nodes — these are the eBPF decisions:"*

- A **green "✓ Connection Permitted"** node means the kernel allowed the call — it shows the hostname and the policy reason (e.g. 'in allowed_hosts').
- A **red "🛡 Attack Blocked by Kernel"** node means the kernel denied it — it shows the hostname and the reason (e.g. 'no policy match').

> *"Above each kernel node there's a small annotation badge — green '✓ Allowed by policy' for safe connections, and red '⚠ Prompt injection detected' for blocked ones. This makes the security decision immediately obvious at a glance without having to read the card detail."*

### 7d. Auto-Follow & Navigation

> *"The graph auto-follows the latest node as new events come in — it smoothly pans and keeps the bottom of the chain centred in view. If you manually pan or zoom, a '↓ Follow latest' button appears in the bottom-right corner — click it to re-engage auto-follow."*

> *"You can also use the built-in zoom controls in the bottom-left to zoom in for detail or zoom out to see the full session at once."*

## 8. The Full Demo Loop

> *"What you're watching right now is a scripted demo session replaying on a loop every ~13 seconds:"*

1. Agent session starts
2. User asks it to fetch `example.com` and summarise it
3. Agent makes a `fetch_url` tool call → kernel allows it → fetch succeeds
4. Agent tries to call `evil.com/exfil` (simulating a prompt injection attack)
5. Kernel intercepts and blocks the connection → alert banner fires → stat rings update → timeline marks the block
6. Agent produces its final answer
7. Session completes — then resets and replays

> *"Every one of those steps is visible simultaneously across the header sparkline, the stat rings, the session flow bar, both event panels, the timeline, and the workflow graph. The whole pipeline — from agent tool call to kernel enforcement — is observable in one place."*

## 9. Key Design Decisions (if asked)

| Decision | Reason |
|---|---|
| WebSocket relay architecture | Decouples the agent runtime from the viewer — senders and viewers connect independently |
| Two-panel split (LLM + Kernel) | Keeps the agent's self-reported events separate from the kernel's ground-truth enforcement |
| Fade-in animation (not slide) | Prevents layout shifts that would reset scroll position in the event panels |
| `overflow-anchor: none` on feed | Stops the browser scroll-anchor heuristic from jumping when new cards are prepended |
| Workflow tab takes full space | The graph needs as much canvas as possible — stat rings are not useful context there |
| Auto-follow with manual override | Lets you inspect past nodes without the view fighting you, with a one-click way back |

The gap between those two layers is exactly where prompt injection and exfiltration attacks live — and this tool makes that gap visible."*
